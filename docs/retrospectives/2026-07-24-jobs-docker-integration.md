# Jobs 接入与 Docker 运行复盘

日期：2026-07-24

本文记录本次异步任务接入和 Docker 运行过程中暴露的问题，重点保留现象、根因、影响、修复方式及后续验证项，供后续迭代复盘。

## 1. 五个 Job 只有定义，没有业务触发入口

### 现象

项目中已经存在以下五个异步任务，但业务流程中没有调用 `dispatch()`，因此正常操作不会产生对应的队列任务：

- `CloseExpiredOrderJob`
- `GenerateApiBillJob`
- `InventoryAlertJob`
- `MerchantWelcomeEmailJob`
- `SyncLogisticsJob`

### 根因

任务类先完成了职责定义，但订单创建、发货、租户创建、库存变化和账单生成等业务入口没有与任务建立连接。代码“存在”不代表能力已经“接入”。

### 影响

- 待支付订单不会按预期自动关闭。
- 库存不足不会触发异步预警检查。
- 新商户不会进入欢迎通知流程。
- 发货后不会触发物流同步。
- API 账单缺少明确、可操作的异步生成入口。

### 修复

将五个 Job 接入对应的业务动作，并在涉及数据库写入时采用事务提交后派发，避免事务回滚但任务已经进入队列：

- 创建订单后延迟派发关单任务，并派发库存检查任务。
- 订单状态变为已发货后派发物流同步任务。
- 创建租户后派发欢迎通知任务。
- 从管理端提供账单生成动作，派发指定租户和账期的账单任务。
- 同时覆盖 API 与 Filament 管理端中实际存在的业务入口。

### 后续验证项

- 验证重复请求、重复点击和任务重试不会造成重复关单、重复账单或重复通知。
- 验证事务回滚时队列中没有残留任务。
- 验证延迟关单任务执行时会再次检查订单状态，已支付订单不得被关闭。
- 为每个派发入口补充队列 fake 测试，并为 Job 的实际处理逻辑补充集成测试。
- 评估库存预警是否需要去重、冷却时间和通知升级策略。

## 2. Compose 缺少 queue worker 和 scheduler

### 现象

应用容器可以提供 Web 请求，但 Compose 原先只有应用、MySQL 和 Redis 服务。即使 Job 已经成功写入 Redis，也没有进程消费；定时任务定义存在，但没有调度进程负责按时触发。

### 根因

队列基础设施只配置了连接，没有配置运行时进程。Laravel 的 Web 服务、队列消费者和调度器是三个不同职责的常驻进程，启动 Octane 不会自动启动后两者。

### 影响

- Redis 中的队列任务持续积压。
- 延迟任务到期后仍不会执行。
- 月度账单、API 用量归集、风险扫描等计划任务不会自动触发。
- 开发环境可能误以为“派发成功”等于“业务已执行”。

### 修复

在 Compose 中增加独立的 queue worker 和 scheduler 服务：

- queue worker 常驻消费 Redis 队列，并配置合理的重试次数和超时。
- scheduler 通过 `php artisan schedule:work` 常驻运行计划任务。
- 两个服务复用应用代码、依赖和环境配置，并等待应用基础环境就绪。

### 后续验证项

- 通过 `docker compose ps` 确认应用、worker、scheduler、MySQL、Redis 均为运行状态。
- 派发一条即时任务和一条延迟任务，确认 worker 均可消费。
- 使用 `php artisan schedule:list` 核对下次执行时间。
- 重启 worker 后验证未完成任务能够继续处理。
- 验证失败任务的记录、重试和人工重放流程。
- 生产环境补充进程存活、队列堆积量、失败任务数和执行耗时告警。

## 3. `schedule:work` 常驻导致日期参数冻结

### 现象

计划任务注册时直接构造 Job，并传入 `now()->subDay()` 计算出的日期。在一次性执行 `schedule:run` 时不明显，但 `schedule:work` 是常驻进程，任务注册结果会长期驻留。

### 根因

日期是在调度进程启动、加载计划定义时计算，而不是在每天真正派发或执行任务时计算。常驻进程不重启时，后续每天可能继续使用启动当天算出的同一个日期。

### 影响

- API 用量归集可能反复处理旧日期。
- 新日期的数据未被归集，导致用量与账单数据不完整。
- 问题具有延迟性，服务刚启动时结果正确，运行数日后才暴露。

### 修复

调度注册时不再传入提前计算的动态日期，由 Job 在实际执行时计算目标日期；或者使用执行时求值的回调。动态业务参数应尽量在任务执行边界生成，而不是在常驻进程启动时固化。

### 后续验证项

- 在不重启 scheduler 的情况下跨日期运行，确认每天处理的目标日期随时间变化。
- 为日期计算注入可控时钟，并添加跨日自动化测试。
- 审查所有计划任务，排查在注册阶段调用 `now()`、`today()` 或读取易变化配置的同类问题。
- 明确补跑历史日期的命令或管理入口，避免复用日常任务的隐式日期逻辑。

## 4. worker/scheduler 并发拉取本地镜像

### 现象

新增 worker 和 scheduler 服务后，执行 Compose 构建和启动时，它们可能尝试从远端仓库拉取应用镜像，而不是稳定复用本地正在构建的镜像。并发启动时尤其容易出现镜像不存在、拉取失败或启动顺序不稳定。

### 根因

多个服务引用了同一个镜像名，但只有应用服务声明了 `build`。Compose 无法可靠地把其他服务视为同一个本地构建目标，可能按“仅引用镜像”的服务处理并尝试 pull。

### 影响

- 首次启动可能失败，且表现依赖本机是否已有旧镜像缓存。
- worker、scheduler 与应用可能使用不同版本的镜像。
- 离线或没有镜像仓库权限时无法启动完整开发环境。

### 修复

为应用、worker 和 scheduler 复用同一个 Compose build 定义，使它们共享同一构建上下文和 Dockerfile；同时统一应用环境、挂载和依赖关系，避免仅靠一个可能需要远程拉取的镜像标签串联服务。

### 后续验证项

- 清理本项目的本地镜像缓存后执行一次完整 `docker compose up --build`，确认不访问不存在的远端应用镜像。
- 修改 Dockerfile 后重新构建，确认三个服务使用相同镜像 ID。
- 在 CI 中执行冷启动验证，避免本地缓存掩盖问题。
- 检查 Compose 配置展开结果，确认三个服务的 build 上下文一致。

## 5. 欢迎邮件与物流同步仍是 mock

### 现象

`MerchantWelcomeEmailJob` 和 `SyncLogisticsJob` 虽已进入队列链路，但当前处理逻辑仍以日志模拟结果，没有调用真实邮件服务或物流平台。

### 根因

项目尚未提供可用的外部服务配置、接口契约、认证信息和适配器实现。现有 Job 主要用于验证异步流程和保留扩展边界。

### 影响

- 商户不会收到真实欢迎邮件。
- 物流状态不会同步到外部承运商，也不会拉回真实运单信息。
- 日志中的“成功”只能表示 mock 逻辑执行完成，不能作为外部业务完成凭证。

### 修复

本次只完成业务入口到队列任务的接入，并明确保留 mock 状态，不把模拟执行描述为真实集成。Job 继续作为外部适配器的调用边界。

### 后续验证项

- 确定邮件供应商、发件域名、模板、退信处理和发送频控。
- 确定物流供应商、下单/查询/取消接口、签名方式和回调协议。
- 为外部调用增加幂等键、超时、重试退避、熔断和失败告警。
- 区分“任务执行成功”“外部平台受理成功”和“最终业务完成”三类状态。
- 在测试环境使用供应商 sandbox 或 fake adapter，禁止自动化测试调用真实生产接口。

## 6. 宿主机直接执行测试无法解析容器数据库地址

### 现象

在宿主机直接执行 `php artisan test` 时，项目 `.env` 中的 `DB_HOST=mysql` 无法被宿主机解析，测试可能长时间等待后报数据库连接错误；同一配置在 Compose 网络内可以正常解析到 MySQL 服务。

### 根因

`mysql` 是 Compose 内部服务名，只在 Docker 网络的 DNS 中有效。宿主机 PHP 进程不在该网络中，却复用了面向容器的环境配置。

### 影响

- 本机测试看似卡住，反馈周期变长。
- 容易把环境连接问题误判为代码或测试问题。
- 不同开发者的宿主机配置会产生不一致结果。

### 修复

本次验证优先在应用容器中执行，让测试进程使用与项目运行环境一致的 PHP 扩展、数据库服务名和网络。宿主机直跑测试需要单独的测试环境配置，例如可解析的 `127.0.0.1`、映射端口或 SQLite 测试库，不能直接照搬容器内 `.env`。

### 后续验证项

- 在容器内运行完整测试套件并记录失败项。
- 为项目约定唯一、清晰的测试入口，并写入 README 或开发脚本。
- 如需支持宿主机测试，增加独立 `.env.testing`，避免修改日常容器配置。
- 检查测试库隔离，确保测试不会连接开发库或生产库。
- 为测试命令增加明确的连接超时，减少错误配置导致的长时间等待。

## 7. 高优先级：测试数据库未与开发数据库隔离

### 现象

`phpunit.xml` 没有显式指定独立的测试数据库。容器内执行 `php artisan test` 时，测试进程会继承 Compose 环境中的 `DB_DATABASE=saas_platform`，即直接连接日常开发数据库。使用 `RefreshDatabase` 的测试会执行迁移重置，可能删除开发库中的表和数据。测试并发执行或两次测试运行时间重叠时，还可能出现迁移竞态，例如“表已存在”或创建外键时被引用表尚不存在。

### 根因

- 测试环境只覆盖了部分 Laravel 配置，没有强制覆盖数据库名和连接参数。
- 容器运行环境与测试运行环境共用同一套数据库变量。
- 缺少在测试启动前校验数据库名称和环境的保护逻辑。
- 多个测试进程可能同时对同一个 schema 执行 `migrate:fresh`、删除表和重建表。

### 影响

- `RefreshDatabase` 可能清空开发数据，属于直接的数据安全风险。
- 正在运行的应用会因表被删除或迁移未完成而报错。
- 并发测试结果不稳定，产生与业务代码无关的迁移失败。
- 如果同类配置被带入其他环境，存在误操作生产数据库的严重风险。

### 修复

必须为自动化测试使用专用数据库，例如 MySQL 的 `saas_platform_testing`，并在 `phpunit.xml`、`.env.testing` 或测试脚本中显式覆盖 `DB_DATABASE`。对于不依赖 MySQL 特性的测试，也可以评估使用独立 SQLite 数据库，以提高隔离性和执行速度。

同时增加强制保护：测试启动时检查当前环境和数据库名称，数据库名不符合测试库命名规则，或疑似连接开发库、生产库时立即终止。CI 和本地测试脚本都应负责创建、迁移和清理专用测试库，不能依赖运行应用的数据库配置。

### 后续验证项

- 在 `phpunit.xml` 或 `.env.testing` 中确认 `DB_DATABASE=saas_platform_testing` 等专用配置真实生效。
- 执行测试前打印或校验实际数据库连接，确认没有连接 `saas_platform` 开发库。
- 在测试启动钩子、CI 和本地脚本中增加“禁止测试连接开发/生产数据库”的硬性检查。
- 运行带 `RefreshDatabase` 的完整测试后，确认开发库表和数据未发生变化。
- 并发测试使用独立数据库、独立 schema 或 Laravel 并行测试的数据库分片机制，避免多个进程重建同一个 schema。
- 增加一次并发或重叠测试验证，确认不再出现表已存在、外键引用表缺失等迁移竞态。
- 将测试数据库的账号权限限制在测试库范围内，从数据库权限层阻止其修改开发库和生产库。

## 8. 低/中优先级：Dockerfile 依赖层耦合导致 Swoole 重编

### 现象

Dockerfile 将系统依赖安装、PHP 内置扩展编译和 PECL Swoole 安装放在同一个 `RUN` 层。即使只是新增 `pdo_sqlite` 这类 PHP 扩展，Docker 也会使整个层的缓存失效，并重新下载、配置和编译 Swoole 6.2.2。Swoole 编译耗时较长，因此冷构建和小幅依赖调整后的重建都需要等待很久。

### 根因

- 变化频率不同的构建步骤被绑定在同一个 Docker 缓存层中。
- Swoole 通过源码编译安装，成本显著高于普通包安装。
- 构建链路依赖实时 PECL 包，缺少稳定的预构建 PHP + Swoole 基础镜像层。
- 依赖版本和缓存策略没有围绕可复现构建与复用进行专门设计。

### 影响

- 本地开发和 CI 的镜像构建反馈变慢。
- 一处无关扩展变更会触发高成本的 Swoole 全量重编。
- CI 冷缓存时资源消耗增加，失败后的重试成本较高。
- 直接依赖远端最新包或未充分固定的构建输入，会降低构建可复现性。

### 修复

本次以功能可运行为主，尚未对 Dockerfile 构建层做完整优化。后续应固定 Swoole 及关键系统依赖版本，并按变化频率拆分或缓存构建步骤，使稳定且昂贵的 Swoole 编译层尽量不受普通 PHP 扩展调整影响。

如果项目会频繁构建，建议维护经过验证的预构建 PHP + Swoole base image，由应用镜像直接继承。基础镜像应有明确版本标签、构建来源和更新流程，避免用不透明的长期浮动标签换取速度。

### 后续验证项

- 固定 Swoole 版本，并确认镜像构建日志中的实际版本与预期一致。
- 调整 Dockerfile 分层后，仅修改应用代码或普通扩展配置，确认 Swoole 编译层能够命中缓存。
- 记录冷构建、缓存构建和 CI 构建耗时，量化优化收益。
- 评估使用 BuildKit cache mount 缓存 PECL、包管理器和编译产物的可行性。
- 如维护 base image，为其增加漏洞扫描、PHP/Swoole 兼容性测试和定期更新机制。
- 确认优化后的镜像仍包含 `pdo_mysql`、`pdo_sqlite`、Redis 与 Swoole 等项目所需扩展。

## 总结

本次暴露的问题集中在“定义”和“真正运行”之间的缺口：Job 类、计划任务和 Redis 配置只是静态结构，只有补齐业务派发入口、worker、scheduler、正确的常驻进程参数计算和稳定的镜像构建链路，异步能力才算完整闭环。同时，外部邮件和物流仍处于 mock 阶段，应继续作为明确的未完成能力跟踪，不能与已完成的队列接入混为一谈。测试数据库隔离问题优先级最高，在隔离和保护措施落地前，不应在共享开发数据库环境中运行带 `RefreshDatabase` 的测试。Docker 构建分层属于低/中优先级效率问题，可在功能链路稳定后通过缓存和预构建基础镜像持续优化。
