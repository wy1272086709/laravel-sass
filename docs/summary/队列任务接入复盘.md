# 队列任务接入复盘

## 复盘范围

本文记录 5 个业务 Job 从“已有类定义”走向“具备触发入口和运行环境”的接入过程，供后续排查、验收和架构复盘使用。

涉及的 Job：

| Job | 业务价值 | 预期触发时机 |
| --- | --- | --- |
| `CloseExpiredOrderJob` | 自动关闭超时未支付订单，释放人工处理成本，避免待支付订单长期占用业务状态 | 订单创建成功后延迟 30 分钟 |
| `GenerateApiBillJob` | 按租户、账期计算 API 用量和超额费用，支持手工重算 | 平台管理员发起指定账期重算，或由月结任务派发 |
| `InventoryAlertJob` | 在库存变化后识别低库存商品，为补货和运营预警提供入口 | 下单扣减库存或商品库存更新后 |
| `MerchantWelcomeEmailJob` | 将租户创建后的欢迎通知移出同步请求，隔离外部通知失败 | 租户创建成功后 |
| `SyncLogisticsJob` | 将发货后的物流同步移出同步请求，便于独立重试和观测 | 订单状态变为已发货后 |

## 发现的问题

### 1. Job 类存在，但业务链路没有 dispatch

接入前，5 个 Job 已经实现了 `handle()`，但业务代码中没有对应的 `dispatch()`。这意味着：

- 单独执行 Job 测试可能通过；
- 线上发生下单、租户创建、发货等业务事件时，Job 不会自动进入队列；
- “代码里有 Job”不能等同于“异步业务已经接通”。

复盘时应同时检查四个环节：业务触发点、队列入队、worker 消费、执行结果记录。只检查 Job 类或 `queue_job_logs` 表并不足以证明链路完整。

### 2. Docker 缺少 queue worker 和 scheduler

应用使用 Redis 队列时，Web 容器只负责接收请求和派发任务，并不会自动执行队列 Job。Laravel Scheduler 也不会因为 `routes/console.php` 中存在调度定义而自动运行。

接入前的 Compose 只有 Web、MySQL、Redis 服务，缺少：

- 常驻 `php artisan queue:work` 的 queue worker；
- 常驻 `php artisan schedule:work` 的 scheduler。

因此，即使业务已经成功 dispatch，任务也会停留在 Redis；月结、API 用量落库、风控扫描等定时任务也不会到点派发。

### 3. 常驻 scheduler 中的日期参数可能冻结

原调度写法在注册任务时计算日期：

```php
Schedule::job(new ApiUsageFlushJob(now()->subDay()->toDateString()))->dailyAt('00:05');
```

`schedule:work` 是常驻进程，这个 `now()` 会在进程启动、加载调度定义时求值，而不是每天 00:05 执行时重新求值。scheduler 连续运行多天后，任务仍可能处理启动当天计算出的旧日期。

修正原则是让 Job 在真正执行时计算默认日期，调度注册只构造 `new ApiUsageFlushJob`；如果必须在调度层传动态值，则使用执行时求值的回调机制。

### 4. Compose 首次启动存在本地镜像并发拉取问题

多个服务依赖同一个应用镜像时，如果只有 Web 服务声明 `build`，worker 和 scheduler 仅引用本地镜像名，首次并发执行 `docker compose up --build` 可能在镜像尚未构建完成时尝试从远端 pull，出现本地镜像找不到或无权限拉取的问题。

修正方向是让三个应用进程共享同一份 build 定义和镜像配置，使 Compose 明确知道该镜像应由本地构建。代码目录、环境变量也应保持一致，避免 Web 能启动但 worker 读取到不同配置。

### 5. 两个外部动作仍然是 mock

`MerchantWelcomeEmailJob` 和 `SyncLogisticsJob` 当前只模拟外部动作并写入任务日志：

- 欢迎邮件尚未接入真实邮件通道、模板和收件地址；
- 物流同步尚未接入物流服务商 API、鉴权、单号映射和回调处理。

`InventoryAlertJob` 当前也以记录低库存统计为主，尚未接入站内通知、短信或其他告警通道。

这类 Job 接入业务触发点后，能够验证异步链路和日志观测，但不能对外宣称真实邮件或物流对接已经完成。

### 6. 本机 PHP 测试会受到容器主机名影响

项目 `.env` 面向 Docker 网络时，数据库主机通常配置为 `mysql`。这个主机名只在 Compose 网络中可解析，本机直接运行 `php artisan test` 时可能持续等待数据库连接，表现为测试卡住或超时。

这不是测试逻辑本身失败。验证时应二选一：

- 在应用容器内执行测试；
- 为本机测试提供独立环境配置，例如可访问的 `127.0.0.1` 数据库端口或 SQLite 测试库。

不要为了让本机命令临时通过而直接改坏容器使用的 `.env`。

### 7. Livewire 测试不要在两次组件交互之间替换队列绑定

测试异步派发时，应优先用 `Bus::fake([目标 Job])` 做窄范围替换。当前项目中使用 `Queue::fake()` 替换整个队列管理器后，Livewire 创建页测试出现已填字段在提交时全部变成必填校验失败；这个失败容易被误判为表单或业务钩子回归。

## 接入边界与设计原则

### 事务提交后再派发

依赖新建或刚更新数据的任务，应在数据库事务成功提交后派发。否则 worker 可能先于事务提交读取数据，或者主事务回滚后仍执行已经入队的任务。

### 状态变化才触发

物流同步应只在订单状态实际变为“已发货”时触发，不能在每次编辑订单时重复派发。超时关单任务执行时也必须再次检查订单状态和创建时间，保证任务可安全重跑。

### 手工重算要有权限边界

API 账单重算会改变计费结果，应由平台管理员发起，并明确租户和账期；不应直接开放给普通租户自行重算应收账单。

### 队列名称要和 worker 能力匹配

当前这些 Job 都使用 Redis 连接的 `default` 队列；`QueueJobLogger` 中的 `billing`、`notification`、`sync_data` 是日志分类，不是队列名。若后续通过 `onQueue()` 拆分队列，必须同步修改 worker 的监听参数，否则部分任务会正常入队却永远不被消费。

## 验证方式

### 静态检查

```bash
php artisan schedule:list
docker compose config
```

确认定时任务已注册，且 Web、worker、scheduler 使用同一个本地构建的应用镜像和一致的运行配置。

### 容器运行检查

```bash
docker compose ps
docker compose logs queue-worker
docker compose logs scheduler
```

确认三个应用进程均持续运行，worker 没有连接 Redis 或数据库失败，scheduler 能按分钟检查调度。

### 业务链路检查

依次验证：

1. 创建订单后，Redis 队列出现延迟关单任务，库存预警任务被派发。
2. 创建租户后，欢迎通知任务被派发并写入 `queue_job_logs`。
3. 将订单改为已发货后，物流同步任务仅派发一次。
4. 平台管理员提交租户和账期后，API 账单重算任务进入 `billing` 队列。
5. 到达调度时间或手动触发调度后，月结、API 用量落库和风控扫描任务由 worker 消费。

自动化测试至少应使用 `Queue::fake()` 断言关键业务动作派发了正确 Job，并补充 Job 本身的幂等与状态判断测试。

## 核心结论

这 5 个 Job 的价值不只是“异步提升性能”，更重要的是把超时关单、计费、库存预警、通知和物流同步拆成可重试、可观测、可独立演进的业务步骤。

完整接入的判断标准是：业务事件能够派发、队列基础设施能够消费、Job 能够幂等执行、结果能够观测、外部适配的真实与 mock 边界清晰。缺少其中任一环节，都只能算局部实现。
